## 2. 프로젝트 목표 및 비즈니스 가치

**고객**
* 새로운 기능을 기획하려고 하는 서비스 기획팀
* 다른 회사의 예시를 바탕으로 서비스 품질을 검토하는 QA팀
* 자료를 얻고자하는 UX 리서치 조직

**판매 대상**
* 카카오톡의 대개편 업데이트로 인한 민심 변화 분석 자료

**기대 효과 (비즈니스 가치)**
* 리뷰를 개편 이전, 개편 직후, 원상복 이후 세 시기로 나누어 비교하면 어떤 이슈는 일시적인 반발이었고, 어떤 이슈는 시간이 지나도 계속 언급되는지를 확인할 수 있을 것이다.
* 이를 통해 서비스 기획 단계에서 과거 UI/UX 개편 사례를 참고하여 **사용자 반발 가능성이 높은 요소를 미리 검토**하고, 업데이트 이후 **어떤 기능을 우선적으로 개선해야 하는지 탐색하는 기초 자료**로 활용될 수 있을 것이다.
* 또한 서비스 기획팀과 QA팀은 시기별 워드클라우드를 통해 **사용자 관심사와 불만이 어떻게 변화했는지를 빠르게 파악**하고, UX 리서치 조직은 **반복적으로 등장하는 키워드를 중심으로 인터뷰나 추가 사용자 조사를 설계**하는 데 활용할 수 있을 것 같다.

**예상되는 한계점**
* 리뷰 작성 시점과 실제 사용 시점이 다를 수 있음
* 워드클라우드는 단어 빈도만 보여줄 뿐 원인까지 설명하지 못함
* 하나의 서비스만으로 일반화하기 어려움

## 3. 데이터셋 구축 (프로토타입)

### 3.1 분석 대상

카카오톡은 2025년 UI를 대규모 개편한 뒤 사용자 반발로 일부 기능을 순차적으로 원상 복구했다.

| 날짜 | 사건 |
|---|---|
| 2025-09-23 | UI 개편 |
| 2025-09-29 | 일부 기능 롤백 |
| 2025-12-15 | 친구 탭 원상 복구 |


이를 기준으로 리뷰를 세 구간으로 나누었다.

| 구간 | 기간 |
|---|---|
| 개편 이전 | 2023-01-01 ~ 2025-09-22 |
| 개편 직후 | 2025-09-23 ~ 2025-12-31 |
| 원상 복구 이후 | 2026-01-01 ~ |

### 3.2 웹 스크레이핑

Google Play 리뷰는 Selenium으로 직접 수집했다.
처음에는 최신순으로 모든 리뷰를 수집하려 했지만, 개편 이후 리뷰가 급증하면서 브라우저 메모리 한계로 중간에 중단되는 문제가 발생했다.
이를 해결하기 위해 다음과 같이 방식을 변경했다.

* 최신순 대신 관련성순으로 수집
* 리뷰 날짜를 기준으로 세 구간으로 자동 분류
* 각 구간 목표 개수를 채우면 자동 종료
* 중복 제거 및 이어받기 기능 추가

또한 폰과 태블릿 리뷰를 각각 수집하여 총 3,000건의 리뷰를 구축했다.

## 5. 프로젝트 결과
![alt text](kakaotalk_reviews_wordcloud.png)
리뷰를 시기별로 비교한 결과, 업데이트 이후 사용자 반응 변화를 탐색하는 데에는 일정 부분 활용 가능성을 확인할 수 있었다.

* 개편 직후에는 메신저(정체성 논란), (카톡의)인스타(화), 목록(친구목록), 롤백 등의 키워드가 크게 나타나 사용자 관심과 불만이 집중된 기능을 빠르게 확인할 수 있었다.
* 원상복구 후에는 또 다른 키워드들이 보이는 모습을 확인할 수 있어 변화를 빠르게 파악할 수 있었다. 따라서 일시적인 이슈와 시간이 지나도 계속 언급되는 이슈를 구분할 수 있었다.
* 모든 리뷰를 직접 읽지 않고도 후속 분석이 필요한 기능을 빠르게 좁혀볼 수 있었다.

반면 워드클라우드만으로는 사용자가 왜 불만을 느꼈는지까지는 설명하기 어려웠으며, 실제 개선 우선순위를 결정하기에는 한계가 있었다.

## 6. 한계 및 개선 방향

**한계**

* 관련성순 수집으로 인해 표본 대표성이 부족할 수 있음
* 긍정 리뷰 수가 적어 일부 워드클라우드 해석이 어려움
* 단어 빈도만으로는 불만의 원인과 중요도를 설명하기 어려움
* 별점으로 분류된 리뷰에 긍정/부정 리뷰가 섞여 있는 경우 많음
* 어플 버전을 알 수 없어 리뷰 작성 내용과 실제 사용 버전이 다를 수 있음
* 폰과 태블릿 리뷰를 함께 분석하여 기기별 특성이 일부 섞였을 가능성

**개선 방향**

* 무작위 또는 전체 리뷰 기반 샘플링으로 대표성 향상
* 딥러닝 기반 모델을 활용한 감정 분석을 통한 고도화된 분석 적용
* 평점, 설치 수, 고객센터 문의량 등 다른 지표와 함께 분석
* 하나의 리뷰 내에서도 긍정/부정 분류

### 8.1. 김민수

* 주제를 더 좁혀야했다. 업데이트 당시의 모든 반응과 변경사항을 담으려 하기보다 가령 친구탭 기능의 업데이트에 한정하고, 얼마나 수정했냐에 따라 기간 사례를 더 쪼개 분석하는게 더 좋았을 것 같다. 다노님이 말하셨던 것처럼 이번 프로토타입은 이번 사례에서만 사용하고, 다른 곳에서 사용하지 못했어야 했을만큼 좁혔어야 했다. 너무 포괄적인걸 담으려했던 게 아닌가 싶다.
* 비즈니스 효과에 집중하다보니 기대효과를 너무 많이 장황하게 많이 적었다. 프로젝트의 한계점을 인식하고 하나만 핀포인트로 제대로 노렸으면 더 좋지 않았을까하다. 고객도 하나에만 집중하는게 좋을 것 같다.
* 무엇보다 결론이 제대로 나지 않았던 것 같다. 워드클라우드에서 나타난 특정 키워드(단어)로 어떤 결론을 낼 수 있었을까?에 대한 고찰이 절대적으로 부족했다. 피드백 당시 다노님이 물어보자 급히 생각할 정도였다. 원상복구 후 보이는 키워드들은 이런게 있다~정도의 정보조차 이 위키에 없었다.ㅜㅜ 구현하는 거에만 집중하지말고 우리가 이걸 한 이유를 생각하고, 제대로 결론을 내려야겠다고 생각했다.
* 주제 브레인스토밍 때 더 많은 것들을 내뱉었다면 좋았을 거 같다.
* 회의 중에는 다노님이 말하신 타임박싱 기법을 적극적으로 활용하지 못한 것 같다.
* 데이터를 모을 때 셀레니움이 느렸다면 다른 방안도 같이 생각해봐야 했다.
* 데이터를 모은 뒤 품질 검사?나 전처리 관련해서도 논의가 부족했던 것 같다. 그냥 모이는대로 확인하지 않고 대충? 써버렸다.
* 위키 글과 통계 자료를 좀 더 보완해야했다. (비율 표시/그래프, 빈도 그래프 등) 워드클라우드만 사용해야한다는 생각에 스스로 결과물에 대한 발전을 시키지 못한 것 같다.(워드클라우드 자체의 한계 때문에)
* 팀 활동 전체 과정을 속기록을 해서 raw 데이터를 모아야 할 것 같다. 이를 통해 우리의 과정을 분석하고, 위키에 각자의 역할, 진행 과정과 시행착오 등을 담는게 좋아보인다. 말로만 하니 하나를 말했더라도 모두가 같은 방향으로 이해를 한 것은 아닌듯 했다.
* 이정도면 되지 않을까라는 생각보다는 스스로가 만족하고 설명을 자신있게 할 만큼 프로젝트의 완성도를 높이는게 중요하다는 생각이 들었다.
* 여담으로 워드클라우드의 가장 큰 차별점과 장점은 시각적 효과인만큼, 역시 빨강 파랑 등의 색깔을 입히는게 더 좋은 것 같다.


### 8.2. 이동찬

* word cloud(일반화하면 특정 툴)에만 매몰되지 않고, 문제가 무엇인지 명확히 파악하고 그걸 해결하기 위한 수단으로 활용할 것. 그 툴을 사용한 이유에 대해 설명 가능할 것
    * 우리가 풀고자 하는 문제(ex. 카카오톡팀의 여론 회복)을 구체적으로 명시 해놓고 항상 볼 수 있도록 큰 글자로 띄워 놓는게 좋을듯
* 왜 이 주제를 선택했는지에 대한 설명이 안됐던 것 같음
    * 이러이러한 문제가 있기 때문에 이런 주제를 갖고 왔다.
    * 1차 프로토타입에서 공통적으로 업데이트 이후 인스타 형식의 숏폼 도입에 불만이 있다는 사실을 알게 되었다.

            * 예를 들면, UI/UX 관점에서의 불편함? 
                * 만약 이게 문제라고 하면 UI/UX의 큰 틀은 유지하되, 숏폼 기능을 부가적으로 도입한 느낌이었다면 소비자의 불만은 없었을까?
            * 숏폼을 도입함으로서 기술적인 문제를 야기한 부분이 있을지?
                * ex) 숏폼 들어오면서 렉이 심해졌다


### 8.3. 최지욱

* 일단 프로젝트에 대한 설명이 부족했다. 어떤 고객들을 대상으로 하는지 구체적으로 설정하는 것이 부족했고 그렇기 때문에 이러한 프로젝트의 필요성을 강조하지 못했다.
* 기대 효과 또한 제대로 된 분석이 이루어지지 않았고 너무 광범위하게 설정하여 명확한 기대 효과를 알 수 없었다.
* 데이터 수집과 분석 과정에서 사용한 데이터가 완전한 특별한 상황을 보여주는 데이터였다.(카카오톡 release & 원상복구) 이러한 데이터에 대해서 분석을 진행하여 이를 일반적인 release 상황에 적용하기 힘들 가능성이 높다는 점을 인지하지 못했다.
* 데이터도 부족했다고 생각한다. 구글 스토어 리뷰 특성상 언제든지 리뷰를 작성할 수 있다. 그 리뷰 작성 날짜가 그 날짜 기준 가장 최신 버전을 사용하고 있다고 대변하지 못한다. 부족한 데이터를 가지고 결론에 끼워 맞추려고 한 것 같다.
* 팀 프로젝트를 진행함에 있어서 그 주에 배운 툴들만 쓰려고 하지 말고 더 확장해 나아가는 필요성을 놓쳤다. 워드 클라우드 이외의 분석 기법들을 활용하여 비지니스 가치를 찾을 수 있도록 노력했어야 했다. 
* 리뷰 한 줄에도 긍/부정 코멘트들이 섞여 있는 것은 이미 알고 있었지만 그 내용들에 대해서도 전처리 할 시도도 하지 않고 쉽게 포기한 것 같다. 
* 프로젝트 과정이 데이터 수집하는데 몰두하다가 다른 내용들을 대부분 놓쳤다. 프로젝트의 본질이 흐려졌고 그 과정에서 데이터에 대한 관심은 대부분 수집 과정에 분포되어있어서 다른 전처리, 데이터 품질 등이 부족했다.
* 프로젝트를 진행한 이후 작성한 위키를 너무 소홀하게 작성했다. 실제로 처음 보는 대상이 우리의 프로젝트 설명글을 읽었을 때 그를 설득할 수 있을 정도로 디테일하고 설득력 있게 작성했어야 한다.


### 8.4 박민서

* 아이디어 선정이나 ‘만들 수 있는 비즈니스 가치’에 대한 좀 더 구체적인 설정과 이유가 필요할 것 같다.
    * 단순 “업데이트 전후 사용자 반응 변화를 비교”가 아니라 비교를 활용해서 "버전 업데이트 전과 후의 리뷰의 긍정 / 부정에 대해 어떠한 점을 개선 혹은 유지할 것인지 추후 업데이트 방향을 제시"와 같은 것으로 제시해야 했던 것 같다.
    * 아이디어 선정 - 분석 - 결과에 끝내지 말고 분석 결과나 결론에서 추가적인 의견이나 제시할 게 있다면 앞의 과정들의 내용을 수정하면서 프로젝트가 구체화되어야 할 것 같다.
    * 결과적으로 Actionable Data / 우리의 글이나 분석 결과를 보고 타겟 대상이 '이걸 해야 하겠구나 / 행동으로 옮겨야 겠구나' 로 전달이 되게 만들어야 할 것 같다.
* 워드 클라우드 속 키워드를 선택했을 때, 그 키워드와 관련한 추가 분석들이 필요해 보였다.
    * 키워드 포함 리뷰에 대한 작은 워드클라우드를 만들면 어떤 이유로 이 키워드가 같이 언급이 되었는지도 말할 수 있을 것 같음
    * 일시적 반발인지 지속적인 반발인지에 대한 시계열 그래프? 트렌드 그래프가 추가적으로 제시되면 해석이나 인사이트 제공에 도움이 될 것 같다.
    * 궁극적으로 앱 버전별로 사용자 리뷰를 수집해서 워드클라우드를 만든다는 생각을 한다고 해도 앱 버전을 수정한다고 했을 때 어떠한 특징들을 수정했고 그에 대한 반응을 볼 것인지 혹은 다른 분석을 할 것인지에 대한 생각은 부족했다.
* (프로토타입) 데이터를 선정할 때에도 기대하는 비즈니스 가치를 정말로 이룰 수 있나?의 관점에 주목해서 선정해야 하고 그 이유도 필요할 것 같다.
* 데이터에 관련한 배경 / 도메인 지식 같은 것도 고려해봤어야 했던 것 같다.
    * 리뷰가 어떻게 수집이 되는지? (사용자만 쓸 수 있는건지 / 모두가 쓸 수 있는건지)

## Docker를 사용하는 이유가 뭘까요?
- 환경 일관성 확보하기 위해 사용한다. 
- "내 PC에서는 되는데 다른 PC에서는 안 된다"는 문제를 이미지 하나로 패키징해서 해결하여 누구나 동일한 조건에서 실행과 배포가 가능하다.

## 어떤 점은 더 불편한가요?
- 번거롭고 불편하다는 의견이 다수 있었다.
- 처음 익히고 연결하는 진입장벽이 크고, 디버깅이나 운영이 복잡해진다. 
- 코드를 조금만 수정해도 이미지를 다시 빌드하고 재배포해야 하는 번거로움이 생긴다. 
- AWS 연동 설정 자체도 복잡하다는 의견 있음

## 만약에 여러대의 EC2에 여러 개의 컨테이너를 배포해야 한다면 어떻게 해야 할까요?
- 오케스트레이터라는 도구를 활용하자는 의견을 모두 제시했다.
- 사람이 서버마다 일일이 실행 및 관리하는 건 비효율적이다.
- 쿠버네티스, ECS 등의 도구로 어떤 EC2에 무엇을 띄울지 자동 결정하고, 컨테이너 장애 시 자동 재시작 등을 처리한다.
