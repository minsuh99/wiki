물리 데이터 모델링: 논리 데이터 모델을 특정 DBMS에 맞는 물리적 요소로 적용하는 과정

* 같은 논리 모델이라도 DBMS마다 타입·문법·기능이 다르다
    * 하나 자주 쓰는걸 정해라.
* 그래서 "관계형 데이터베이스에 대한 지식"이 필요하다
* 결과물은 우리가 매일 쓰는 그것 → 테이블, 컬럼, PK, FK, 인덱스
논리 모델 = 무엇을 저장할까 / 물리 모델 = 어떻게 저장할까

+) 검증 - 제일 쉬운 방법: 데이터 몇 개 넣어보기 (실제 데이터와 유사한 분포의 데이터)

물리 모델링 절차 (큰 그림)
전환은 보통 이 순서로 진행됩니다.
단계	하는 일
1. 테이블 매핑	엔티티 → 테이블, 단위·우선순위 결정
2. 컬럼 매핑	식별자 → PK, 일반 속성 → 컬럼, 관계 → FK
3. 상세화	무결성 · 인덱스 · 반정규화 · 저장기법
+) 클러스터링, 샤딩, 파티셔닝..?

테이블 매핑
- 엔티티 -> 테이블
- 서브타입의 고민
* [회원]  (슈퍼타입)
*   ├─ 개인회원  (서브타입)
*   └─ 법인회원  (서브타입)

전략	테이블 구성
Roll-Up	하나로 통합 (슈퍼+서브 전부 한 테이블)
Roll-Down	서브타입마다 따로 (슈퍼는 각 테이블에 흡수)
Hacksaw	슈퍼 1개 + 서브 N개 (각각 분리)
어떤 조회가 많은가로 결정합니다.


Roll-Up — 하나의 테이블로 통합
- 슈퍼타입 이름으로 테이블 하나, 구분자 컬럼으로 서브타입을 나눕니다.

CREATE TABLE MEMBER (
  member_id   BIGINT       PRIMARY KEY,
  member_type CHAR(1)      NOT NULL,   -- 'P'개인 / 'C'법인 (구분자)
  name        VARCHAR(50)  NOT NULL,
  birth_date  DATE,                    -- 개인만 사용
  biz_no      VARCHAR(20)        -- 법인만 사용
);


장점	단점
조회가 간단, 조인이 줄어 속도 유리	서브타입별 NOT NULL·기본값을 일관되게 못 줌
복잡한 로직을 하나의 SQL로 처리	컬럼이 많아져 저장공간 증가 가능
서브타입 구분 없이 전체 조회 쉬움	항상 구분자로 걸러야 함
새 서브타입 추가 시 값만 추가	일부 서브타입만 봐도 전체 스캔 비용 발생
적용 → 서브타입 간 동질성이 크고, 같이 조회하는 일이 많을 때 (결국 사용해 봐야 함)

Roll-Down — 서브타입마다 테이블
- 슈퍼타입 컬럼을 각 서브타입 테이블에 복사해서 따로 만듭니다.

CREATE TABLE INDIVIDUAL_MEMBER (
  member_id  BIGINT       PRIMARY KEY,
  name       VARCHAR(50)  NOT NULL,    -- 슈퍼타입 속성
  birth_date DATE         NOT NULL     -- 개인 전용
);
CREATE TABLE CORPORATE_MEMBER (
  member_id  BIGINT       PRIMARY KEY,
  name       VARCHAR(50)  NOT NULL,    -- 슈퍼타입 속성
  biz_no     VARCHAR(20)  NOT NULL     -- 법인 전용
);

장점	단점
서브타입별 조회 시 구분 불필요	둘을 같이 보려면 UNION 필요 → 성능 저하
가져올 컬럼이 명확	통합 SQL 작성이 어려움
NOT NULL·기본값을 제대로 지정 가능	전체 식별자(UID) 유지 관리 어려움
테이블 크기가 작아 풀스캔에 유리	합친 VIEW는 조회만 가능
적용 → 서브타입별로 따로 처리하는 업무가 많을 때

Hacksaw — 슈퍼 + 서브 각각
- 공통은 슈퍼타입 테이블에, 전용 컬럼은 서브타입 테이블에. FK로 연결.
CREATE TABLE MEMBER (              -- 슈퍼타입
  member_id BIGINT PRIMARY KEY,
  name      VARCHAR(50) NOT NULL
);
CREATE TABLE INDIVIDUAL_MEMBER (   -- 서브타입
  member_id  BIGINT PRIMARY KEY,
  birth_date DATE   NOT NULL,
  FOREIGN KEY (member_id) REFERENCES MEMBER(member_id)
);

장점	단점
통합·분리의 장점을 동시에 수용	같이 처리하려면 항상 조인 발생
슈퍼타입만 조회 시 SQL 간단	전체 통합 조회 많으면 과다 조인
컬럼 중복 없음 → 저장공간 절약	SQL이 한층 복잡해질 수 있음
슈퍼/서브 업무 중요도가 대등할 때 적합	슈퍼↔서브 데이터 동기화 필요
적용 → 공통(슈퍼) 트랜잭션이 많고, 서브타입 처리도 독립적으로 자주 발생할 때


결국, "실제로 어떻게 조회할 건가?"
* 거의 같이 조회한다 → Roll-Up (조인 없이 한 방에)
* 거의 따로 조회한다 → Roll-Down (각자 깔끔하게)
* 둘 다 중요하다, 공통 처리가 많다 → Hacksaw
실무에서 자주 마주치는 의사결정입니다."정답"이 아니라 트레이드오프를 이해하는 게 핵심.

컬럼 매핑
논리 모델의 식별자는 물리 모델에서 키가 됩니다.
논리 모델	물리 모델
실질식별자(UID)	Primary Key
보조식별자	Unique Index
복합 식별자	복합 PK (컬럼 순서 검토)
복합 키는 자주 쓰는 조회 패턴에 맞춰 앞 컬럼 순서를 정하면, 인덱스 효율이 좋아집니다. (우선순위??)

	Primary Key	Unique Index
목적	제약조건 + 인덱스	인덱스
NULL	허용 안 됨	허용됨
개수	테이블당 1개	여러 개 가능
식별자 인식	가능 (리버스 시)	불가능
참조 무결성(FK)	지정 가능	—
- 이런걸 다 고려하면서 만들기는 처음엔 굉장히 어렵다..

—> 식별자는 PK로!
성능 차이는 없습니다. PK를 쓰는 게 이득.
* 무결성 오류 방지 — 참조 무결성(FK) 지정 가능
* PK 컬럼엔 NULL 불가 → 빈 식별자가 원천 차단
* 모델을 되살릴(리버스) 때 식별자를 명확히 인식
+) Unique Index는 "PK는 아니지만 중복은 막고 싶은" 보조 식별자에 사용.


컬럼 매핑
- 한글 속성명을 실제 컬럼 정의로
    - 고객명 -> cust_name VARCHAR(50)
    - 가입일자 -> join_date DATE
    - 가입경로 -> join_channel CHAR(3)
+) 괄호 안의 숫자는 한국어 byte 크기가 아닌 그냥 글자수

* 단어 사전: 한글 → 영문명 통일 (다음 시간 "데이터 표준화"에서!)
* 도메인 정의: 타입과 길이 결정 (VARCHAR(50)? INT? DECIMAL?)


컬럼 NULL 허용 여부 — 신중하게
NULL을 허용할지는 데이터의 활용 형태를 보고 정해야


데이터 무결성 (Integrity)
데이터의 정확성·일관성·신뢰성을 잘못된 갱신으로부터 지키는 장치.
종류	의미
실체 무결성	모든 행은 식별자(PK)를 가지고, NULL 없이 유일
참조 무결성	FK 값은 부모 PK에 존재하거나 NULL
도메인 무결성	타입·길이·CHECK·NOT NULL·DEFAULT·코드값
사용자 정의 무결성	업무 규칙 (트리거 등)

+) 참조 무결성 - FK 삭제 옵션
FOREIGN KEY (member_id) REFERENCES MEMBER(member_id)
    ON DELETE CASCADE       -- 부모 삭제 시 자식도 함께 삭제
    -- ON DELETE SET NULL    -- 자식 FK를 NULL로
    -- ON DELETE RESTRICT    -- 자식이 있으면 부모 삭제 거부

요새는 잘 안 쓰고 ‘is_delete’ 컬럼 (True / False)을 활용해 soft deletion으로 구현 (복구 빠르게, 안전하게)

인덱스 설계 (기초)
인덱스 = 데이터를 빨리 찾는 지름길
* PK, FK엔 인덱스가 자동/관례적으로 생긴다
* 자주 조회 조건(WHERE) 에 쓰이는 컬럼에 인덱스를 고려
* 자주 같이 쓰이는 컬럼은 결합 인덱스 (순서가 중요!)
* 분포도(중복 정도)가 낮은 컬럼일수록 효과적
    * 성별 같은 경우(특히 분포가 극단적인 경우) 효과 X

CREATE INDEX idx_orders_cust ON ORDERS (cust_id);
- 인덱스는 조회는 빠르지만 쓰기는 느려짐. -> 너무 많이 만드는것도 문제
- 그래도 많이 만들거나 만들어진 이유 -> 조회가 중요해서
