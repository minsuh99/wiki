식별자
- 테이블에서 인스턴스를 유일하게 구분할 “속성”이 필요

조건: 유일성 (값이 겹치지 않는다.) + 최소성 (더 줄이면 유일성이 깨진다.)
* 유일성만 있고 최소성이 없으면 → 군더더기가 붙은 키 (제거할 필요가 있음)
* 둘 다 있어야 → 깔끔한 "후보키"

￼

포함관계: 슈퍼키 ⊇ 후보키 ⊇ (기본키 ∪ 대체키)

슈퍼키: 유일성만 만족, 최소성은 상관 X
Ex) 학생 테이블에서
- { 학번 }                    → 유일 O  (군더더기 없음)
- { 학번, 이름 }            → 유일 O  (이름은 군더더기)
- { 학번, 이름, 학과 }    → 유일 O  (다 군더더기)

후보키: 유일성 + 최소성 만족
Ex) 학생 테이블에서
-   { 학번 }         → 유일 O, 최소 O  → 후보키 ✅
-   { 이메일 }      → 유일 O, 최소 O  → 후보키 ✅
-   { 학번, 이름 } → 최소 X              → 후보키 ❌ (슈퍼키일 뿐)
    * 한 테이블에 후보키가 여러 개 있을 수 있다
    * 이 중 하나를 대표로 뽑으면 → 기본키 (사용자 마음대로 설정 가능)

* 기본키(Primary Key, PK): 후보키 중 대표로 선정한 하나
* 대체키(Alternate Key): 후보키인데 기본키로 안 뽑힌 나머지

후보키 = { 학번 }, { 이메일 }
   학번    →  기본키(PK)로 선정 ✅
   이메일 →  대체키 (여전히 유일하지만 대표는 아님)
* 대체키는 보통 DB에서 UNIQUE 제약으로 살려둔다 (unique 인덱스 설정, 데이터 정합성 유지 / 공간 차지, insert 성능 저하 [딱히 체감되는 정도는 아님])
	- unique index를 운영 중 중간에 삽입되면 DB가 다운..된다..

외래키 (FK)
* 다른 엔티티의 기본키를 참조하는 속성
* 관계(1:N 등)를 테이블에서 표현하는 수단
* ￼
    * FK 덕분에 "이 주문이 누구 것인지" 추적 가능
    * 존재하지 않는 회원번호는 못 넣게 막아줌(참조 무결성)
    * "참조하는 데이터는 반드시 존재해야 한다” (참조된 부모 테이블에)
    * 부모 테이블의 키값 or NULL 가능
    * FK 자체에 중복으로 들어갈 수도 있어서 중복 인덱스가 자동으로 들어감?(확인 필요)

+) 보조식별자
* 기본키 말고 인스턴스를 식별할 수 있는 또 다른 속성
* 주로 다른 엔티티와 연결고리 역할이 필요할 때 정의
* 실무에서는 UNIQUE INDEX 로 구현되는 경우가 많다


자연키 vs 대리키
구분	본질식별자 (자연키)	인조식별자 (대리키)
영어	Natural Key	Surrogate Key
정체	업무상 의미가 있는 값	시스템이 인위적으로 만든 값
예시	주민번호, 이메일, 학번	auto_increment ID, UUID
의미	데이터 자체에 뜻이 있음	그냥 일련번호, 뜻 없음
+) UUID (무작위 문자열, 8a463aa4-b1dc-4f27-9c3f-53b94dc45e74) -> 정보보안을 위함 + 분산 시스템일 때 중복된 데이터가 있을 확률이 거의 0에 수렴, unique 보장 / 저장공간을 너무 많이 차지함 (16바이트) + 정렬[특히나 시간순, v7에서 해결]이 안 되어 있음)


본질식별자 (자연키)
* 내용(의미)상의 진정한 식별자
* 집합의 본질을 설명하는 "의미상의 주어" — 누가/무엇을/언제 [옛날 얘기?]
* 특별한 하자가 없으면 이게 곧 기본키가 된다
보험계약 엔티티의 본질식별자
  누가(WHO)    → 고객번호
  무엇을(WHAT) → 보험상품코드
  언제(WHEN)   → 계약일자
  → 이 셋의 조합이 한 건의 계약을 구분
* 장점: 의미가 있어 직관적
* 문제: 자주 깨진다

자연키가 깨지는 순간들
본질식별자가 "논리적으로는" 맞아도 현실에서 문제가 생긴다.
* 유일성 실패: 동일 고객·동일 상품·동일 일자에 동시 가입? → 셋을 합쳐도 유일하지 않다
* 너무 김 / 복잡함: 키가 3~4개 컬럼이라 자식 테이블마다 다 끌고 다님 → JOIN 조건이 매번 길어진다
* 값이 바뀜: 이메일·전화번호·학번처럼 나중에 변경될 수 있음 → PK가 바뀌면 참조하던 FK가 전부 흔들림
 -> 이럴 때 등장하는 것이 인조식별자(대리키)


인조식별자 (대리키, Surrogate Key
* 본질식별자를 대체하려고 인위적으로 만든 식별자
* 의미 없는 값 — 그냥 유일하기만 하면 된다
>>> CREATE TABLE contract (
	 contract_id BIGINT AUTO_INCREMENT PRIMARY KEY,  -- 인조키
	customer_no VARCHAR(20),
	product_code VARCHAR(20),
	contract_date DATE,
  UNIQUE (customer_no, product_code, contract_date) -- 자연키는 UNIQUE로 보존
);
* 흔한 형태: BIGINT AUTO_INCREMENT, UUID
* 자연키는 버리지 말고 UNIQUE 제약으로 살려두는 게 좋다.


인조키를 쓰는 이유 — 핵심 4가지
#	원칙	무슨 상황?
1	유일성 보장	자연키만으로 유일성이 안 될 때
2	편의성·단순성	자연키가 길어 JOIN이 복잡할 때
3	변경 대비	자연키 값이 바뀔 수 있을 때 (이메일 등)
4	자손이 많을 때	자식 테이블이 많아 키를 계속 상속시킬 때
"왜 인조키?"를 물으면 위 4가지 중 하나로 답할 수 있어야 한다
￼
* 모든 테이블에 id 박는 것도 정답은 아니다
* 다만 실무 BE 기본값은 인조키(PK) - 변경에 가장 안전하기 때문

인조키 주의 사항
* 불러줄 대상이 없으면 인조키 PK는 불필요한 인덱스만 늘린다
    * 아무도 그 id로 조회하지 않는데 만들어두는 경우
* 자연키 자체가 짧고 안정적이면(예: 국가코드 KR, US) 그대로 PK가 나을 수도
* 인조키를 쓰더라도 자연키의 유일성은 반드시 UNIQUE로 지켜라
    * 안 그러면 같은 회원이 두 번 가입하는 중복이 슬그머니 들어온다

인조키는 "편하려고" 쓰는 것이지, 데이터 정합성을 포기하려고 쓰는 게 아니다

CREATE TABLE member (
  member_no  BIGINT AUTO_INCREMENT,   -- 인조 기본키
  email      VARCHAR(255) NOT NULL,   -- 자연키 → 대체키
  name       VARCHAR(50)  NOT NULL,
  PRIMARY KEY (member_no),            -- 기본키
  UNIQUE KEY ux_email (email)         -- 대체키(보조식별자)
);

CREATE TABLE orders (
  order_no   BIGINT AUTO_INCREMENT PRIMARY KEY,
  member_no  BIGINT NOT NULL,
  FOREIGN KEY (member_no) REFERENCES member(member_no)  -- 외래키
);
