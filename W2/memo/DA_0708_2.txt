정규화

이상현상 (정규화가 안 되어 있어 발생하는 문제)
- 삽입 이상: 넣고 싶지 않은 데이터까지 억지로 넣어야 하는 문제
    * "아직 수강신청은 안 했지만 신규 학생 333을 등록하고 싶다"
    * 그런데 이 표는 과목코드가 있어야 행이 생긴다 (PK 일부)
    * 결국 가짜 과목 정보를 넣거나, 등록 자체가 불가능
학번	과목코드	성명	...
333	???	신입생	신청 안 했는데 과목을 넣어야 함
- 갱신 이상: 하나의 사실을 바꾸기 위해 여러 행을 동시에 고쳐야 하는 문제

	학생 111이 개명해서 "김수현 → 김철수"가 됐다고 합시다.
학번	과목코드	성명
111	E	김수현 → 김철수
111	M	김수현 → ??? (빠뜨리면?)
* 한 행만 고치고 다른 행을 빠뜨리면 → 111이 두 사람이 됨
* 데이터 정합성 붕괴

- 삭제 이상: 지우고 싶지 않은 정보까지 함께 사라지는 문제

학생 222가 "과학" 수강을 취소해서 그 행을 지웁니다.
학번	과목코드	성명	과목명
222	S	임영웅	과학
* 수강 취소만 하려 했는데, "과학"이라는 과목 정보 자체가 사라질 수 있음
* 하나의 행에 여러 주제가 섞여 있어서 생기는 일


그래서, 정규화
* 위 3가지 이상현상의 공통 원인 = 중복
* 중복의 원인 = 한 테이블에 서로 다른 주제의 데이터가 섞여 있음 
    * ERD를 잘 짜면 해결됨
* 정규화(Normalization): 속성 간 종속 관계를 따져 큰 테이블을 여러 작은 테이블로 무손실 분해하는 과정
* "무손실" = 다시 조인하면 원래대로 복원 가능


함수 종속성 (Functional Dependency)
X → Y : X를 알면 Y가 하나로 정해진다
* 읽는 법: "Y는 X에 종속된다", "X가 Y를 결정한다"
* X = 결정자(Determinant), Y = 종속자(Dependent)

예시:
* 학번 → 성명 : 학번을 알면 이름이 정해진다 ✅
    * 학번이 달라지면 이름이 바뀌나? (그건 아님)
    * 일대일 함수 뭐 이런거 아님, 그냥 함수가 만족되면 돼.
    * 하나의 x에 대해 하나의 y가 대응
* 과목코드 → 과목명 : 과목코드를 알면 과목명이 정해진다 ✅
* 학번 → 과목명 : 학번만으론 과목명을 못 정한다 ❌

복합키 {학번, 과목코드}를 기준으로 봅니다.
종류	의미	예시
완전 함수 종속	키 전체가 있어야 정해짐	{학번, 과목코드} → 평가코드
부분 함수 종속	키의 일부만으로 (다른 속성이)정해짐	학번 → 성명
이행 함수 종속	A -> B, B -> C 이면 A -> C인 경우 + 종속성이 정해짐	평가코드 → 평가내역

학번	성명	수강과목	과목명	평가코드	평가내역
111	김수현	E, M	영어, 수학	A, C	최우수, 보통
222	임영웅	E, S	영어, 과학	B, D	우수, 미흡
* 수강과목 등 한 칸에 여러 값이 들어가 있음 (반복 그룹)
* PK도 명확하지 않음

제1정규형: 모든 속성은 원자값을 가져야 한다. (중복, 반복 그룹 제거)
학번	과목코드	성명	과목명	평가코드	평가내역
111	E	김수현	영어	A	최우수
111	M	김수현	수학	C	보통
222	E	임영웅	영어	B	우수
222	S	임영웅	과학	D	미흡
* PK = {학번, 과목코드} (복합키) +) 대체키로 복합키를 없앨 수도 있음

제1정규형이 남긴 문제
* 성명 김수현이 학번 111의 행마다 반복됨
* 원인: 성명은 과목코드와 상관없이 학번만으로 정해진다
* 즉 부분 함수 종속 (학번 → 성명) 이 살아있음 → 2NF로 제거

제2정규형: 부분 함수 종속 제거 [복합키일 때 발생 가능성]
{학번, 과목코드}에서 종속 관계를 따져보면:
* 학번 → 성명 (학번만으로 결정) → 부분 종속, 떼어낸다
* 과목코드 → 과목명 (과목코드만으로 결정) → 부분 종속, 떼어낸다
* {학번, 과목코드} → 평가코드 → 완전 종속, 남긴다

학생 (PK: 학번)
학번	성명
111	김수현
222	임영웅
과목 (PK: 과목코드)
과목코드	과목명
E	영어
M	수학
S	과학
수강 (PK: {학번, 과목코드})
학번	과목코드	평가코드	평가내역
111	E	A	최우수
111	M	C	보통

* 성명·과목명 중복이 사라졌습니다 ✅
* 그런데... 평가코드와 평가내역 사이가 수상합니다 (종속성의 Loop?)
    * {학번,과목코드} → 평가코드 → 평가내역

제3정규형: 이행 종속 제거
수강 (PK: {학번, 과목코드}, FK: 평가코드)
학번	과목코드	평가코드
111	E	A
111	M	C
222	E	B
222	S	D
* 평가내역을 "최우수 → 매우우수"로 바꿔도 평가 테이블 한 줄만 수정

-> 비정규화된 테이블을 정규화된 테이블 4개로 분해

전제: 한 교수는 한 과목만 담당한다고 합시다.
학번	과목	담당교수
111	영어	김교수
222	영어	이교수
111	수학	박교수
PK = {학번, 과목}, 종속 관계는 두 가지
    * {학번, 과목} → 담당교수 (완전 종속)
    * 담당교수 → 과목 (한 교수 = 한 과목이므로)
* 키가 아닌 속성끼리의 종속은 없으니 3NF는 만족
* 그런데 "영어"가 교수마다 반복되고 있습니다...

BCNF: 모든 결정자(→의 왼쪽)는 후보키(unique + 최소성)여야 한다
* 담당교수 → 과목 : 담당교수는 결정자인데 후보키가 아님 → BCNF 위반
* 그래서 3NF인데도 이상현상이 생깁니다
    * 이교수의 담당 과목이 바뀌면 여러 행을 고쳐야 함 (갱신 이상)
    * 수강생 없는 교수의 담당 과목은 저장 불가 (삽입 이상)
교수 (PK: 담당교수) / 수강 (PK: {학번, 담당교수}) 로 분해:
담당교수	과목
김교수	영어
이교수	영어
학번	담당교수
111	김교수
222	이교수
* 3NF를 만족하면 대부분 BCNF도 자동으로 만족합니다
    * 위반은 "복합키 + 후보키가 아닌 결정자" 조합에서 드물게 등장
    * 그래서 BCNF를 "3.5정규형" 이라고도 부릅니다
* 단, 위 분해는 {학번, 과목} → 담당교수 제약을 한 테이블 안에서 강제하지 못하게 됨 — BCNF의 대가
* 실무 감각: 3NF까지 정확히 하고, 이런 패턴이 보이면 BCNF를 의심


* 응집도 높은 엔티티 도출: 관련 속성끼리 모이고 무관한 건 분리됨
* 중복 최소화: 이상현상이 사라지고 저장 효율도 좋아짐
* *무결성 극대화*: 사실을 한 곳에서만 관리 → 정합성 보장
* **주 식별자 정확성 향상**: 종속성 분석으로 올바른 PK가 드러남
* 확장성 증대: 업무 변경 시 영향 범위가 명확하고 작음
Vs
* 테이블이 잘게 쪼개졌다 = 데이터를 다시 합치려면 조인(JOIN) 필요
* 학생 이름 + 과목명 + 평가내역을 함께 보려면 4개 테이블 조인
* 읽기가 매우 빈번한 경우, 조인 비용이 부담될 수 있음
